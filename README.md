## 2. Relationship Between the Files

These 4 files are not independent. They form a small analysis project with clear dependencies.

### 2.1 `q1.py`
This is the **core simulation model**.

It builds a hybrid continuous-time smartphone battery model and is responsible for the base simulation itself.

Main capabilities:

- Battery SoC evolution
- Temperature evolution
- Radio state machine (`IDLE / TAIL / ACTIVE`)
- Foreground workload modeling
- Power modeling for screen, radio, GPS, Bluetooth, background wakeups, etc.
- Thermal feedback and interaction/coupling terms
- Output of full time-series simulation trajectories

You can regard `q1.py` as the **underlying simulation engine**.

---

### 2.2 `plot_sim_output.py`
This is the **visualization utility**.

It reads the `sim_output.csv` generated by `q1.py` and produces PNG figures.

Main plots:

- SoC / terminal voltage / current curves
- Temperature / throttling-factor curves
- Power breakdown figure
- Radio state timeline

You can regard `plot_sim_output.py` as the **result plotting tool**.

---

### 2.3 `q2.py`
This is a **second-layer analysis script** built on top of `q1.py`.

It is used for:

- Multi-scenario comparison
- Battery-life difference analysis
- Driver attribution
- First-order sensitivity analysis
- Batch export of scenario results
- Batch plotting

It first tries to import `final_unified_model`; if that fails, it falls back to `q1.py`.
In the current file set, it will normally use `q1.py`.

You can regard `q2.py` as the **batch experiment script for scenario comparison and attribution analysis**.

---

### 2.4 `q3.py`
This is a **third-layer statistical analysis script** built on top of `q1.py`.

It is used for:

- OAT (one-at-a-time) local sensitivity analysis
- Monte Carlo global sensitivity analysis
- Spearman rank-correlation analysis
- Hypothesis/assumption-variant testing
- Paired bootstrap confidence intervals
- MannŌĆōWhitney U tests

You can regard `q3.py` as the **statistical experiment script for stability and robustness analysis**.

---

## 3. Recommended Order of Use

A practical order for understanding and running the project is:

1. Run `q1.py` first  
   Generate the base simulation CSV outputs.
2. Run `plot_sim_output.py`  
   Plot the output produced by `q1.py`.
3. Run `q2.py`

4. Run `q3.py`

## 5. Detailed Description of Each File

---

## 5.1 `q1.py`

### 5.1.1 Purpose
`q1.py` is the core modeling file of the whole project. It simulates how smartphone battery charge (SoC) and temperature evolve over time under realistic usage conditions.

It is not just a simple linear battery-drain model. It combines:

- a continuous-time differential model,
- a discrete state-machine model,
- and a multi-subsystem power coupling model.

---

### 5.1.2 Main Data Structures

#### `BatteryParams`
Defines battery-related parameters, such as:

- rated capacity `Q_nom_Ah`
- nominal voltage `V_nom`
- baseline internal resistance `R0_ohm`
- capacity health factor
- resistance health factor
- OCV range
- power-path efficiency `eta_dc`

#### `ThermalParams`
Defines thermal-model parameters, such as:

- ambient temperature `T_amb_C`
- thermal capacitance `C_th_J_per_C`
- cooling coefficient `h_W_per_C`
- throttling start temperature
- full-throttling temperature
- minimum performance factor

#### `ScreenParams`
Defines screen-related parameters, such as:

- OLED / LCD-like behavior
- screen size
- refresh rate
- resolution scaling
- baseline screen power
- nonlinear brightness exponent

#### `SoCComputeParams`
Defines upper power bounds for CPU / GPU / ISP / NPU / DRAM / storage related computation.

#### `SoCArchParams`
Defines SoC architecture differences, such as:

- process node
- hardware video decoding capability
- ISP capability
- NPU capability

#### `InteractionParams`
Defines explicit coupling terms, such as:

- screen ├Ś rendering
- weak signal ├Ś background ├Ś radio
- temperature ├Ś leakage

#### `RadioParams`
Defines radio state-machine parameters, such as:

- tail time constant
- active hold time
- state-dependent power
- weak-signal penalty
- Wi-Fi / cellular switching cost

#### `SensorParams`
Defines GPS and Bluetooth-scan power usage.

#### `OSParams`
Defines system-level behavior such as background wakeups and scheduler aggressiveness.

#### `ModelParams`
Packages all the above parameter groups into one unified model-parameter object.

#### `ScenarioStep`
Defines one simulation input step, including:

- time step length
- screen state
- brightness
- refresh rate
- foreground task type
- network request rate
- signal quality
- network type
- GPS state
- Bluetooth scan state
- background wake rate
- ambient temperature

#### `SimState`
Defines the evolving simulation state:

- current time `t_s`
- current SoC `z`
- current temperature `T_C`

---

### 5.1.3 Main Functions

#### `ocv_from_soc`
Computes open-circuit voltage (OCV) from SoC and temperature.

#### `q_eff_Ah`
Computes effective battery capacity considering temperature and health.

#### `r_int_ohm`
Computes internal resistance considering temperature and health.

#### `throttle_factor`
Returns a performance factor from temperature to model thermal throttling.

#### `RadioRRC`
A radio state-machine class that models:

- `IDLE`
- `TAIL`
- `ACTIVE`

and transitions between them.

#### `screen_power_W`
Computes screen power consumption.

#### `workload_utilization`
Maps a foreground scenario into CPU / GPU / ISP / NPU / storage utilization.

Supported foreground types:

- `video`
- `game`
- `camera`
- `navigation`
- `social`
- `idle`

#### `compute_power_W`
Computes computation-side power and accounts for:

- thermal throttling
- process-node efficiency
- hardware acceleration paths
- scheduling aggressiveness

#### `background_wake_power_W`
Converts background wake frequency into average power.

#### `solve_current_A`
Solves discharge current from the required power.

#### `simulate`
The main simulation loop of the entire model, producing the full trajectory.

#### `load_steps_from_csv`
Reads simulation input from an external CSV scenario file.

#### `built_in_demo_scenario`
Generates a built-in demo scenario.

#### `summarize`
Extracts SoC at a specified checkpoint and the predicted depletion time from simulation results.

#### `save_csv`
Saves the simulation trajectory to CSV.

## 5.4 `q3.py`

### 5.4.1 Purpose
`q3.py` focuses on higher-level stability, robustness, and hypothesis-testing analysis, making it suitable for the ŌĆ£third questionŌĆØ section of a report or paper.

Typical questions it addresses:

- Which parameters is the model most sensitive to?
- How large is the uncertainty in the outputs?
- Do different modeling assumptions significantly change the conclusions?
- What does the output uncertainty range look like?

---

### 5.4.2 Main Analysis Modules

#### 1) Baseline Scenario Analysis
A baseline scenario is selected and run once as the reference for later comparisons.

#### 2) OAT Local Sensitivity Analysis
Key internal parameters are perturbed one at a time by ┬▒delta to estimate first-order sensitivity.

#### 3) Monte Carlo Global Sensitivity Analysis
Internal parameters and/or external scenario parameters are assigned uncertainty distributions and then sampled in batch experiments.

#### 4) Spearman Rank-Correlation Sensitivity
Uses SpearmanŌĆÖs rho to rank monotonic relationships between parameters and outputs.

#### 5) Assumption-Variant Testing
Compares the baseline with multiple variants such as:

- `baseline`
- `no_interaction`
- `aggressive_throttle`
- `longer_radio_tail`

#### 6) Statistical Tests
The script implements:

- paired bootstrap confidence intervals
- bootstrap p-values
- MannŌĆōWhitney U tests

## 8. Known Notes and Caveats

### 8.1 Import logic in `q2.py` / `q3.py`
Both scripts first try:

```python
import final_unified_model as m
```

and if that fails, they fall back to:

```python
import q1 as m
```

Therefore, even if `final_unified_model.py` is not present, the scripts can still run as long as `q1.py` is available.

---

### 8.2 Known issue with `q3.py --help`
`q3.py` contains a help string with a raw `%` character.
On some Python versions, running:

```bash
python q3.py --help
```

may trigger an `argparse` formatting error.

This does **not** affect the main analysis functionality of the script, but it can affect help-text display.

A typical fix is to replace `%` with `%%` in the relevant help string.

---

### 8.3 Fonts and Chinese plotting labels
The plotting scripts try to configure a Chinese-capable font automatically.
If your system does not have an appropriate Chinese font installed, you may see:

- garbled Chinese characters
- missing-glyph warnings
- fallback to a default font

This usually does not affect numerical results, but it may affect figure appearance.

---

### 8.4 Some scenarios may not deplete within `max-hours`
For low-power scenarios such as standby, if `--max-hours` is too small, the results may show:

- `terminated_reason = max_time`
- `t_empty_h = NaN`

This does not mean the program is incorrect; it only means the simulation hit the time limit before the battery was exhausted.

---

### 8.5 Per-scenario plotting in `q2.py` can be time-consuming
If many scenarios are used and each scenario exports its own figures, runtime can increase noticeably.

Ways to speed things up:

- use `--no-plots`
- use `--quick`
- use a larger `--dt`
- disable per-scenario figure export with `--no-per-scenario-figures`
